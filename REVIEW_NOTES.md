# Review Notes — Twitter Stream Analytics (Event-Driven Microservices)

Consolidated notes covering all applicable master-prompt docs. This is by far the largest project reviewed this session (14 Maven modules: platform services, streaming ingestion, dual query stacks, shared libraries) - comparable in scope to NEXUS itself, which the rest of this review series used as a reference. Full NEOBANK-depth review of every module wasn't feasible in one pass; this covers the highest-leverage findings and states explicitly what wasn't covered, rather than silently sampling.

## IMPORTANT: pre-existing uncommitted work found, left untouched

`git status` showed ~140 files already modified/deleted in the working tree before this pass started (pom.xml files across several modules, `WebSecurityConfig.java` in elastic-query-service, `KafkaStreamsRunner.java`, several docker-compose YAML files, some deleted test files and a deleted `keycloak_postgres.yml`/`bootstrap.yml`). Some of this is recognizably my own earlier work in this repo (the CORS/frontend-panel comment already in `elastic-query-service`'s `WebSecurityConfig.java`). I did not revert, overwrite, or build on top of any of it beyond the specific fixes documented below - reviewed the code as it currently sits in the working tree, same as every other project this session.

## Fixed: real, already-triggered bug — literal `${HOME}` directories

`docker-compose/.env` and `docker-compose/dev-env.sh` had `KAFKA_STREAMS_STATE_FILE_LOCATION=${HOME}/kafka-streaming-state`. `.env`-style files are parsed literally (no shell expansion) by docker-compose, IDE run configs, and most dotenv loaders - only an actual `source`'d bash script expands `${HOME}`. This had already fired for real: two literal directories named `${HOME}` (not expanded) existed in the checkout, containing hundreds of tracked Kafka Streams RocksDB state-store files (`.sst`, `LOCK`, `MANIFEST`, etc.) - pure regeneratable local cache data that should never have been committed. Fixed the root cause (both files now use a portable relative path, `./kafka-streaming-state`), added a `.gitignore` entry, and removed the ~145 tracked garbage files from git and disk (confirmed via content inspection that this was 100% regeneratable RocksDB cache, not anything with informational value worth preserving - unlike the AWS-LAMBDA-SERVERLES project's tfstate files earlier this session, which I only untracked, not deleted).

## Fixed: Kafka consumer had no error handler (shared library, affects every consumer built on it)

`kafka/kafka-consumer`'s `KafkaConsumerConfig.kafkaListenerContainerFactory()` had no `CommonErrorHandler` set - every `@KafkaListener` across every service built on this shared module (kafka-to-elastic-service, analytics-service) fell back to Spring Kafka's own default behavior for a listener that throws, with no deliberate bounded-retry or DLQ policy. Same class of gap found and fixed in FOOD-ORDERIN and GYM_MOSTER earlier this session. Added a `DefaultErrorHandler` with a bounded `FixedBackOff` (3 retries, 1s apart) that logs and moves on rather than retrying forever or silently dropping - the generically-correct default for a shared library that doesn't know each service's DLQ topic naming convention. A real per-service DLQ topic is the natural next step.

## Fixed: gateway-service's circuit breaker and rate limiter were fully built but never wired to any route

This was the most interesting finding in the whole repo: `GatewayConfig.java` already defines a Resilience4j circuit-breaker customizer *and* a `KeyResolver` bean (`authHeaderResolver`) meant for per-user rate limiting; `pom.xml` already pulls in `spring-cloud-starter-circuitbreaker-reactor-resilience4j` and `spring-boot-starter-data-redis-reactive`; `FallbackController` already implements three ready-made fallback endpoints (`/fallback/query-fallback`, `/fallback/analytics-fallback`, `/fallback/streams-fallback`) *with its own test file*. None of it was ever referenced by a route filter in `config-server-repository/gateway-service.yml` - every building block for edge resilience existed and none of it ran. Added `CircuitBreaker` (pointing at the matching existing fallback URI) and `RequestRateLimiter` (10 req/s replenish, burst 20, keyed on the `Authorization` header via the existing `authHeaderResolver` bean) filters to the `elastic-query-service`, `analytics-service`, and `kafka-streams-service` routes. Left `elastic-query-web-client`'s route unchanged - it has no matching `FallbackController` endpoint and is a server-rendered Thymeleaf client, not a JSON API a generic fallback model would suit.

## Flagged, not fixed: `SERVICES.md`'s "gateway... integrated with... Keycloak for auth" overstates what the gateway itself does

`gateway-service`'s `WebSecurityConfig` has `.anyExchange().permitAll()` with CSRF disabled - the gateway itself performs **no** authentication. This is not a raw hole, though: verified that `elastic-query-service` and `analytics-service` (and per `services.yml`'s env vars, `kafka-streams-service` too) each independently run as full OAuth2 Resource Servers with audience-validated JWTs (`AudienceValidator`, one per service, each checking a distinct custom audience claim) and method-level `@PreAuthorize`/`@PostAuthorize` checks - a request without a valid, correctly-scoped JWT is rejected at the actual protected endpoint regardless of what the gateway does. This is a legitimate "zero-trust, auth-at-every-service" architecture, not a broken one. What's actually wrong is the documentation: `SERVICES.md` reads as if the gateway is the auth chokepoint, when it's really just a router with edge-level resilience (now) but no auth of its own. Not fixed by adding gateway-level JWT validation - that's a real architectural choice (redundant defense-in-depth vs. keeping auth solely at each service) that deserves a deliberate decision, not a unilateral change bolted onto a review pass. Worth a documentation correction at minimum.

## Confirmed already correct

- Every Kafka producer path (`ENCRYPT_KEY`-style secrets aside) is centrally configured with `acks: all` and `retry-count: 5` in `config-server-repository/application.yml` - combined with the Spring Boot 3.1 / Kafka 3.4+ client defaults, this is effectively idempotent producer behavior without needing an explicit `enable.idempotence=true` override.
- `ENCRYPT_KEY` (the one secret that actually matters for this stack - it's Spring Cloud Config's property-encryption key) is *never* hardcoded anywhere across any docker-compose file; every service pulls it from `${ENCRYPT_KEY}`, sourced from `.env`'s placeholder value.
- Every consumer path has real Kafka consumer resilience settings externalized (session timeout, heartbeat interval, max poll interval/records) rather than hardcoded magic numbers - `KafkaConsumerConfigData` makes all of it tunable per-environment via Config Server.
- The two query-service pairs (blocking Spring MVC + Thymeleaf vs. reactive WebFlux) genuinely share their model/index-client libraries (`elastic-query-service-common`, `elastic-query-web-client-common`) rather than duplicating logic - a real shared-library split, not copy-paste twins.
- Swagger/OpenAPI annotations (`io.swagger.v3.oas.annotations`) are already present on every controller method reviewed (`AnalyticsController`, `ElasticDocumentController`, `KafkaStreamsController`) with per-status-code `@ApiResponses` - relevant context if a future Swagger/OpenAPI documentation pass targets this project, since the annotation groundwork already exists.

## Flagged, not fixed (lower priority, local-dev-only exposure)

`KEYCLOAK_PASSWORD=admin`, `DB_PASSWORD=keycloak` (Keycloak's own bootstrap admin + backing Postgres), `GF_SECURITY_ADMIN_PASSWORD=admin` (Grafana), and `POSTGRES_PASSWORD=postgres` are hardcoded directly in `services.yml`/`servicesAdvance.yml`/`keycloak_authorization_server.yml`/`monitoring.yml`/`postgres.yml`. All four are the well-known, standard quickstart defaults for their respective images (not unique secrets), and this stack has no evidence of being deployed anywhere beyond local Docker Compose - much lower severity than the AWS-LAMBDA-SERVERLES project's real, already-deployed credential leak earlier this session. Not changed given this project's much larger overall scope already consumed this pass's budget; if this stack is ever deployed somewhere reachable, these four need to move to required env vars with no default, same treatment as `ENCRYPT_KEY` already gets.

## Not reviewed this pass (explicit, not a silent gap)

Given the scale (14 modules), the following were **not** reviewed in this pass and are flagged rather than silently skipped:
- `kafka-streams-service`'s actual Streams topology logic (word-count business logic itself, `KafkaStreamsRunner.java` - which shows as pre-existing-modified in git status, so it may already be mid-change).
- `analytics-service`'s Postgres/JPA persistence layer and Avro-to-entity transformation logic in depth (Doc 5 SQL territory).
- `elastic-query-service`/`reactive-elastic-query-service`'s actual Elasticsearch query-building code (Doc 7 NoSQL territory) beyond confirming the shared-library structure.
- `config-server`/`discovery-service` internals beyond confirming they're standard Spring Cloud Config/Eureka setups.
- The `servicesAdvance.yml` HA topology's specific behavior under actual failover.
- Full read of all 18 existing test files across the repo - spot-checked `FallbackControllerTest.java` only (confirmed it exists and matches the fallback controller it's testing).
- Dockerfile hardening (Doc 3): not applicable - this project builds images via `mvn spring-boot:build-image` (Cloud Native Buildpacks), no Dockerfile exists anywhere in the repo to review.
